# NAS 服务器数据结构普查

> 本仓库唯一的说明文档，负责：**工作交接**、**项目说明**、**改进说明**、**遗留问题与续跑指引**。
> 除本文件外，其余 `.md` 都是脚本自动生成的**成果报告**（数据），不是文档。

---

## 一、这是什么（项目说明）

对一台 NAS 服务器做**目录结构普查**，把每个共享盘的文件夹结构、文件数量、体量摸清楚，产出可读的结构报告，供后续数据盘点、清理、下载补全使用。

- **数据源**：SMB 网络共享盘 `//hmy@192.168.0.102/<项目名>`，在 macOS 上挂载于 `/Volumes/<项目名>`。
- **产物**：每个项目一份「目录结构报告」`*.md`；扫描器另外维护「树状态 JSON」（断点续扫用）和「不可读清单 TXT」。
- **难点**：SMB 盘在长时间扫描中会**反复掉线/僵死**，天真的扫描会把「掉线」误当成「目录不存在」而砍掉整棵子树，甚至用坏结果静默覆盖好结果。当前使用的 v3 扫描器就是为解决这些事故而硬化过的（详见「改进说明」）。

**当前状态：7 个项目全部扫描完成 ✅**（2026-07-21 智慧工地续扫补齐后收官）。全盘零不可读、零中断、零丢子树；唯一的量化保留项是智慧工地里 6 个超大图片/标注目录按设计只统计到文件数下限，见第三节。

## 四、扫描器用法速查

当前脚本：`scripts/_目录结构_快速版_v3.py`（纯标准库，`python3` 直接跑，无需安装依赖）。在**本地 macOS**上、连上 NAS 后使用。

```bash
# 全量扫描一个新盘（生成 md + 树状态 json + 不可读清单）
python3 scripts/_目录结构_快速版_v3.py "/Volumes/<项目名>" <输出目录>

# 断点续扫（只重扫失败子树，合并回主结构；幂等，可反复跑）
python3 scripts/_目录结构_快速版_v3.py --resume <项目名>_树状态.json <输出目录>

# 彻底模式（关掉「文件堆」抽样，逐个核实，更准但更慢）
python3 scripts/_目录结构_快速版_v3.py "/Volumes/<项目名>" --thorough

# 只读复查不可读清单（不重建子树）
python3 scripts/_目录结构_快速版_v3.py --recheck <项目名>_不可读清单.txt <输出目录>

# 播种历史高水位基线（事故后先把已知好数据写进去，守卫据此拒绝低质覆盖）
python3 scripts/_目录结构_快速版_v3.py --seed-baseline <项目名> <文件夹数> <文件数> <输出目录>
```

> 三点务必注意：
> 1. **输出目录要落本地磁盘**，别写到 SMB 盘上（掉线会连输出一起丢）。
> 2. 扫描目标写 macOS 挂载点 `/Volumes/<项目名>`，脚本靠「卷根」`/Volumes/<项目名>` 做掉线探测。挂载：`open "smb://hmy@192.168.0.102/<项目名>"`。
> 3. `Ctrl-C` 会把已扫部分安全收尾并标成「可 --resume 续扫」，中断不丢进度。

### 若某个超大目录想拿到精确文件数
默认按设计截断即可。确需精确时，对该目录单独跑一次 `--thorough`，并临时调大脚本顶部的 `ENUM_TIME_CAP` / `ENUM_CAP`（否则仍会到点截断）。一般不建议——纯图片堆数几十万个只换来一个更精确的数字，不划算。

## 五、改进说明（脚本演进 & 后续建议）

### 已做的硬化（v1 → v3，均来自实际事故复盘）
- **v1 `_目录结构生成器.py`**：自适应折叠 + 汇总卡片，产出好看，但对网络掉线毫无防护。
- **v2 `_目录结构_全展开版.py`**：只 `readdir`、不 `stat` 大小，百万文件也能几分钟扫完；但同样不抗掉线。
- **v3 `_目录结构_快速版_v3.py`（当前）** 针对一次真实事故（2026-07-13，一次扫描把 268 万文件塌成 43 万、还静默覆盖了 2.5 小时的好结果）做了三大硬化：
  1. **抗掉线 + 抗僵死**：`scandir` 失败时先探「卷根还健不健康」给 `ENOENT` 消歧；判为掉线就**阻塞等重连再重读同一目录**，永不因掉线丢子树。目录 I/O 跑在受监视子线程里，SMB 僵死（syscall 发出去不返回、Ctrl-C 都杀不掉）时主流程能靠轮询脱身。
  2. **防覆盖 + 灾难守卫**：写前带时间戳备份 + 原子写；一旦本次结果比历史高水位骤降（<50%）、或有顶层目录不可读、或扫描目标消失，就判「疑似不完整」，大声告警并另存 `SUSPECT`，**绝不覆盖**既有好结果。（智慧工地那次中断正是它兜住的。）
  3. **断点续扫**：以「树状态 JSON」为持久状态，`--resume` 只重扫标记为不可读的子树并原地合并，不必从头重扫整盘。

### 后续可考虑的改进（非必须）
- **概览计数小瑕疵**：续扫后 md 概览里「超大目录/触发安全上限」两项只反映本轮重扫（如智慧工地显示 3、实为 6）。可把它们也改成从整棵树现算（像 `文件夹/文件/不可读` 那样），与 JSON 对齐。
- 语料库盘的「未下载完整清单」当时未随全盘扫描一起更新，可在方便时补一次以刷新 `_疑似未下载完整清单.md`。
- 若经常要对同一盘扫描，建议先 `--seed-baseline` 播种一次已知好水位，进一步降低误覆盖风险。
