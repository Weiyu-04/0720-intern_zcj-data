# NAS 服务器数据结构普查

> 本仓库唯一的说明文档，负责：**工作交接**、**项目说明**、**改进说明**、**遗留问题与续跑指引**。
> 除本文件外，其余 `.md` 都是脚本自动生成的**成果报告**（数据），不是文档。

---

## 一、这是什么（项目说明）

对一台 NAS 服务器做**目录结构普查**，把每个共享盘的文件夹结构、文件数量、体量摸清楚，产出可读的结构报告，供后续数据盘点、清理、下载补全使用。

- **数据源**：SMB 网络共享盘 `//hmy@192.168.0.102/<项目名>`，在 macOS 上挂载于 `/Volumes/<项目名>`。
- **产物**：每个项目一份「目录结构报告」`*.md`；扫描器另外维护「树状态 JSON」（断点续扫用）和「不可读清单 TXT」。
- **难点**：SMB 盘在长时间扫描中会**反复掉线/僵死**，天真的扫描会把「掉线」误当成「目录不存在」而砍掉整棵子树，甚至用坏结果静默覆盖好结果。当前使用的 v3 扫描器就是为解决这些事故而硬化过的（详见「改进说明」）。

> 命名约定：扫描器对项目 `X` 会成套产出 `X_目录结构.md` / `X_树状态.json` / `X_不可读清单.txt`。
> 后缀带 `SUSPECT-<时间戳>` 的，是「防覆盖守卫」判定本次结果**疑似不完整**时的另存版本——它**不会覆盖**任何既有好结果，等挂载稳定后用 `--resume` 续跑即可。

## 四、⚠ 智慧工地项目 —— 如何续跑（本地终端命令）

**背景**：上次用「快速版 v3」扫描智慧工地盘时中途中断，防覆盖守卫把半成品另存成了 `SUSPECT` 文件（没覆盖任何好结果）。待补的是 32 个目录：`report_video` 下 24 个按日期的子目录 + 8 个顶层目录（`吊装视频`、`基坑临边视频`、`高空作业视频`、`王超-备份` 等）。续扫**只会重扫这 32 个子树**并原地合并，不会从头重扫整盘。

> 前提：你在**本地 macOS**上、且能连上这台 NAS（脚本按 macOS 的 `/Volumes/...` 挂载点工作）。

**① 挂载 NAS**（把共享盘挂到 `/Volumes/智慧工地项目`，这是 JSON 里记录的扫描目标）：

```bash
open "smb://hmy@192.168.0.102/智慧工地项目"
```

Finder 弹窗输入密码后会自动挂载。确认挂载成功：

```bash
ls "/Volumes/智慧工地项目"          # 能列出内容就 OK
```

**② 断点续扫**（在仓库根目录执行；输出仍写回 `智慧工地项目_待续扫/`，与状态文件同处）：

```bash
python3 scripts/_目录结构_快速版_v3.py \
  --resume "智慧工地项目_待续扫/智慧工地项目_树状态.SUSPECT-20260717-175433.json" \
  "智慧工地项目_待续扫"
```

跑完后会在 `智慧工地项目_待续扫/` 得到：
- 若 32 个目录都补齐且守卫通过 → **正式版** `智慧工地项目_目录结构.md` + `智慧工地项目_树状态.json`（此时可把它们挪到 `reports/` 和 `state/`，并删掉 `SUSPECT` 半成品）。
- 若仍有目录读不了 → 再落一份新的 `SUSPECT`，且会刷新 `智慧工地项目_不可读清单.txt`。**续扫是幂等的**，可以反复跑：

```bash
# 反复续扫，直到不可读清单清空（每次都对着最新的 树状态 json 跑）
python3 scripts/_目录结构_快速版_v3.py \
  --resume "智慧工地项目_待续扫/智慧工地项目_树状态.SUSPECT-20260717-175433.json" \
  "智慧工地项目_待续扫"
```

**可选：先只读复查**（不重建子树，只快速看看那 32 个目录现在能不能读，判断 NAS 是否已稳定）：

```bash
python3 scripts/_目录结构_快速版_v3.py \
  --recheck "智慧工地项目_待续扫/智慧工地项目_不可读清单.txt" \
  "智慧工地项目_待续扫"
```

**可选：某个目录实在太大卡住**，可用 `--skip` 明确跳过它（树里会标注，不影响其余）：

```bash
python3 scripts/_目录结构_快速版_v3.py \
  --resume "智慧工地项目_待续扫/智慧工地项目_树状态.SUSPECT-20260717-175433.json" \
  "智慧工地项目_待续扫" \
  --skip "/Volumes/智慧工地项目/事件图片-视频/report_video/2025-10-04"
```

> 中断了也不怕：`Ctrl-C` 会把已扫部分安全收尾并标成「可 --resume 续扫」，下次接着跑。

## 五、扫描器用法速查

当前脚本：`scripts/_目录结构_快速版_v3.py`（纯标准库，`python3` 直接跑，无需安装依赖）。

```bash
# 全量扫描一个新盘（生成 md + 树状态 json + 不可读清单）
python3 scripts/_目录结构_快速版_v3.py "/Volumes/<项目名>" <输出目录>

# 断点续扫（只重扫失败子树，合并回主结构）
python3 scripts/_目录结构_快速版_v3.py --resume <项目名>_树状态.json <输出目录>

# 彻底模式（关掉「文件堆」抽样，逐个核实，更准但更慢）
python3 scripts/_目录结构_快速版_v3.py "/Volumes/<项目名>" --thorough

# 只读复查不可读清单（不重建子树）
python3 scripts/_目录结构_快速版_v3.py --recheck <项目名>_不可读清单.txt <输出目录>

# 播种历史高水位基线（事故后先把已知好数据写进去，守卫据此拒绝低质覆盖）
python3 scripts/_目录结构_快速版_v3.py --seed-baseline <项目名> <文件夹数> <文件数> <输出目录>
```

> 两点务必注意：
> 1. **输出目录要落本地磁盘**，别写到 SMB 盘上（掉线会连输出一起丢）。
> 2. 扫描目标写 macOS 挂载点 `/Volumes/<项目名>`，脚本靠「卷根」`/Volumes/<项目名>` 做掉线探测。

## 六、改进说明（脚本演进 & 后续建议）

### 已做的硬化（v1 → v3，均来自实际事故复盘）
- **v1 `_目录结构生成器.py`**：自适应折叠 + 汇总卡片，产出好看，但对网络掉线毫无防护。
- **v2 `_目录结构_全展开版.py`**：只 `readdir`、不 `stat` 大小，百万文件也能几分钟扫完；但同样不抗掉线。
- **v3 `_目录结构_快速版_v3.py`（当前）** 针对一次真实事故（2026-07-13，一次扫描把 268 万文件塌成 43 万、还静默覆盖了 2.5 小时的好结果）做了三大硬化：
  1. **抗掉线 + 抗僵死**：`scandir` 失败时先探「卷根还健不健康」给 `ENOENT` 消歧；判为掉线就**阻塞等重连再重读同一目录**，永不因掉线丢子树。目录 I/O 跑在受监视子线程里，SMB 僵死（syscall 发出去不返回、Ctrl-C 都杀不掉）时主流程能靠轮询脱身。
  2. **防覆盖 + 灾难守卫**：写前带时间戳备份 + 原子写；一旦本次结果比历史高水位骤降（<50%）、或有顶层目录不可读、或扫描目标消失，就判「疑似不完整」，大声告警并另存 `SUSPECT`，**绝不覆盖**既有好结果。
  3. **断点续扫**：以「树状态 JSON」为持久状态，`--resume` 只重扫标记为不可读的子树并原地合并，不必从头重扫整盘。

### 后续可考虑的改进（非必须）
- 续跑收尾：智慧工地补全后，把正式版 `目录结构.md`/`树状态.json` 归入 `reports/`、`state/`，删掉 `SUSPECT` 半成品，让结构进一步收敛。
- 语料库盘的「未下载完整清单」当时未遍历完，可在方便时补一次全盘复核以更新 `_疑似未下载完整清单.md`。
- 若经常要对同一盘扫描，建议先 `--seed-baseline` 播种一次已知好水位，进一步降低误覆盖风险。
